# caker — Agent 协作约定

本文件约束在本仓库（`caker`）上工作的 AI Agent 行为。

## 1. 改代码前须征得同意

在修改本仓库内任何文件（或用户点名的其他路径）前，必须先说明**拟改动的文件、改动要点、原因**；征得用户**明确同意**后，再执行编辑或会改变仓库/运行状态的命令。

## 2. 默认不改代码

用户未明确授权「可以改 / 帮我改进仓库」等时，只提供说明、方案或可复制片段；**不**擅自改文件、**不**擅自提交。

## 3. 范围聚焦本仓库

协作与实现讨论默认**只围绕 `caker`**。其他项目仅作极简背景或类比时可一笔带过；**不**展开其他仓库的实现细节，也**不**把跨仓改动当作本任务的一部分，除非用户明确要求。

## 4. 发出回复前的自检

在**准备输出每条用户可见消息前**，须结合**当前对话上下文与用户约束**（含本文件及与本任务相关的仓库规则），对拟将发出的内容**具体评估一遍**是否**正确、合理**（事实是否成立、推理是否自洽、是否满足用户前提、是否违背上文约定）；若有任一项不成立，**先改正再发出**，不得为省事跳过该步骤。

## 5. 进入下一阶段：先提交，再规划并贴代码

当用户明确表示**推到下一个阶段 / 进入下一阶段 / 开始下阶段**等（自然语言等价表达亦可，意图为「当前里程碑结束，启动下一里程碑」）时，视为对**仅为 checkpoint 目的**的 `git commit` 的授权；须按顺序执行：

1. **先落盘到 Git**：查看 `git status`；若有可提交的改动，先完成提交。**提交说明**须写清本阶段**全部**落盘改动：标题一句话概括里程碑或主题；正文按文件或模块列出要点（路由、状态、节点、配置、规则文档等），避免只用模糊总述一句。若工作区干净或无可提交项，向对方说明后再继续。若存在不属于本阶段的改动、冲突或敏感文件，须先列出并请用户决定，**不**强行提交。

2. **再规划并贴代码**：完成上述步骤后，须给出下一阶段的**目标、要点、验证方式**，且**必须**严格依据 `docs/agent_skills_build_guide.md` 中**该下一里程碑**的「文件清单与分工」与「关键代码骨架」，按跟写指南中的代码骨架格式**贴出全部相关代码**，不得漏表中所列文件。贴码格式须满足：**每个文件块标题含「新建/续写」与「分工：我写｜你手敲｜一起写」**（与指南 `[我写]` / `[你手敲]` / `[一起写]` 一一对应），**每个代码块下方附解说**。指南标为「你手敲」的段落仍须**完整贴出**供用户抄写，不得省略为「请自行实现」。**贴代码不等于已获权修改仓库**；实际编辑文件仍须遵守 §1。

3. **改仓库须另授权**：除第 1 步中的 checkpoint 提交外，用户未在同一轮或后续消息中明确授权「可以改 / 帮我改进仓库」时，**不**在同一轮贴码之后擅自批量改文件；若用户明确授权，可在贴码后继续实现。
